# Event Catalog

This catalog defines the asynchronous contract between the Phase 4 microservices.
Events and commands are transported through RabbitMQ. Each service owns its
database and communicates through these messages instead of reading another
service's data store.

## Message Envelope

Every event and command must use the same envelope shape:

```ts
export interface MessageEnvelope<TPayload> {
  messageId: string
  messageType: string
  messageVersion: number
  correlationId: string
  causationId?: string
  occurredAt: string
  producer: 'work-order-service' | 'billing-service' | 'execution-service'
  payload: TPayload
}
```

| Field | Rule |
| --- | --- |
| `messageId` | Globally unique ID for idempotency. Consumers must store processed IDs before applying side effects. |
| `messageType` | Stable message name, for example `WorkOrderReadyForQuote`. |
| `messageVersion` | Integer schema version. Start at `1`; increment only for incompatible payload changes. |
| `correlationId` | End-to-end trace ID. Preserve the same value across the whole Saga. |
| `causationId` | Optional ID of the message that caused this message. |
| `occurredAt` | ISO-8601 timestamp generated by the producing service. |
| `producer` | Service that produced the message. |
| `payload` | Versioned business payload. It must not require database reads from another service. |

## Naming

- Commands use imperative intent: `StartExecution`, `RequestBillingCompensation`.
- Events use past-tense business facts: `QuoteGenerated`, `PaymentConfirmed`.
- Message names are written in English and must match the ubiquitous language in code.
- Routing keys use kebab-case: `work-order.ready-for-quote`, `payment.confirmed`.

## Idempotency

RabbitMQ messages can be redelivered. Every consumer must be idempotent.

Minimum consumer behavior:

1. Read `messageId`.
2. Check whether the message was already processed.
3. If processed, acknowledge the message and do not repeat side effects.
4. If not processed, perform the local transaction and record `messageId` atomically with the side effect whenever the database supports it.
5. Publish follow-up messages only after the local transaction succeeds.

Provider idempotency keys, such as Mercado Pago keys, are separate from
`messageId`. Billing must persist both the local message ID and the provider
idempotency key.

## Retry And Failure Handling

- Transient failures may be retried by the consumer.
- Permanent business failures must publish explicit failure events.
- Unknown provider events, especially payment webhooks that arrive before the
  local payment exists, must be parked or retried instead of discarded.
- After the retry budget is exhausted, publish or record a manual-review outcome.

## Messages

### `WorkOrderReadyForQuote`

| Property | Value |
| --- | --- |
| Kind | Event |
| Producer | `work-order-service` |
| Consumer | `billing-service` |
| Routing key | `work-order.ready-for-quote` |
| Version | `1` |
| Purpose | Ask Billing to generate a quote automatically from an immutable Work Order snapshot. |

```ts
interface WorkOrderReadyForQuotePayload {
  workOrderId: string
  customerId: string
  vehicleId: string
  items: Array<{
    itemId: string
    type: 'REPAIR_SERVICE' | 'PART' | 'SUPPLY'
    name: string
    quantity: number
    unitPriceInCents: number
    totalPriceInCents: number
  }>
  totalAmountInCents: number
}
```
